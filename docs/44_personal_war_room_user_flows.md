# 个人作战室 V1 用户工作流

## Flow 1：每天打开作战室

用户意图：

用户想知道：今天我最该关注什么？

页面路径：

进入个人作战室首页。

关键显示字段：

- 当前阶段
- 当前核心目标
- 当前主要风险
- 今日最重要军师提醒
- 今日唯一最重要动作
- 高价值信号列表

系统展示：

- 当前局势
- 军师提醒
- 今日动作
- 高价值信号

成功标准：

用户在 30 秒内知道今天最该做什么，以及为什么不该先做别的事。

失败状态：

- 第一屏只出现聊天框
- 第一屏只出现历史列表
- 没有今日动作
- 没有风险或门槛
- 用户仍需要向系统解释“我现在在哪个阶段”

## Flow 2：查看为什么被提醒

用户意图：

用户看到军师提醒后，想知道系统为什么这么判断。

页面路径：

个人作战室首页 -> Advisor Brief -> 提醒详情。

关键显示字段：

- `signal.description`
- `evidence_chain`
- `touchpoint.reason`
- `touchpoint.recommended_action`
- `touchpoint.consequence_if_ignored`
- `counter_argument`

系统展示：

`signal -> evidence_chain -> touchpoint -> recommended_action -> consequence_if_ignored`

成功标准：

用户能判断提醒是否有证据、是否有反方判断、是否值得行动。

失败状态：

- 只展示提醒文本
- 没有证据链
- 没有反方判断
- 没有不行动后果
- 推荐动作无法执行或过于空泛

## Flow 3：检查是否可以进入下一阶段

用户意图：

用户想确认当前是否可以推进到下一阶段。

页面路径：

个人作战室首页 -> Commitments & Gates。

关键显示字段：

- 当前阶段
- 当前阶段门槛
- 已满足条件
- 未满足条件
- 禁止事项
- 用户确认状态

系统展示：

- 当前阶段门槛
- 已满足条件
- 未满足条件
- 禁止事项

成功标准：

用户能明确知道是否允许进入下一阶段；如果不允许，知道卡在哪里。

失败状态：

- 系统直接建议进入下一阶段但不给门槛
- 忽略禁止事项
- 把用户冲动当成阶段确认
- 没有显示未满足条件

## Flow 4：复盘一个模型假设

用户意图：

用户想检查系统对自己的理解是否可靠。

页面路径：

个人作战室首页 -> Personal Model Hypotheses -> hypothesis 详情。

关键显示字段：

- 原假设
- 当前假设内容
- 置信度
- 置信度变化
- `revision_explanation`
- 证据
- 反例
- 后续验证计划

系统展示：

- 原假设
- 置信度变化
- 模型修正原因
- 后续验证

成功标准：

用户能判断这条个人模型假设是否应该被保留、降权、继续观察或人工纠正。

失败状态：

- 只显示假设结论
- 看不到修正原因
- 置信度变化没有解释
- 反例缺失
- 后续验证计划空泛

## Flow 5：从当前局势跳转历史审计

用户意图：

用户想看某条判断的完整历史链路。

页面路径：

个人作战室首页 -> Audit Entry -> `history.html`。

关键显示字段：

- 当前提醒或信号关联的 case
- history 面板入口
- `phase_context`
- `evidence_chain`
- `revision_explanation`
- `audit_readiness`

系统跳转：

跳转到 `history.html` 对应 case 或历史面板。

成功标准：

用户能从当前提醒追溯到历史审计链路，确认判断不是黑箱。

失败状态：

- 没有 history 面板入口
- 无法关联 case
- 只能看到历史，不能回到当前局势
- 审计字段缺失
