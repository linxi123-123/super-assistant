# V1 语义簇规则层

## 目标

第二轮复测证明：原句有效，但变体失败。问题不是链路，而是识别层仍在匹配句子表面。

本次新增 `semanticClusters`，把规则从单句关键词升级为语义簇：

```text
基础事件分类
-> semantic cluster 匹配
-> indicators 命中计分
-> 当前阶段相关度加权
-> 行动价值加权
-> 达到阈值生成 signal
-> 关键 cluster 强制生成 touchpoint
```

## 计分方式

每个 cluster 按以下因素计算：

- `indicators` 命中数量
- 当前阶段相关度
- 事件类型加权
- 行动价值
- negative examples 命中惩罚

当前阈值：

```text
score >= 4
```

关键簇命中后必须触达：

- `judgment_quality_risk`
- `platform_competition_risk`
- `scope_creep_risk`
- `commitment_gate`

## Cluster 1：judgment_quality_risk

语义本质：

用户担心系统只是在复述、改写或模板化分析，而没有理解用户为什么在意、当前处境和真实判断链路。

语义线索：

- 关键词 + 生成
- 真正理解 + 处境
- 看起来 + 分析
- 没有真正 + 抓住
- 为什么 + 在意
- 换一种说法 + 输出
- 复述 + 输入
- 模板 + 提醒
- 只是 + 总结
- 整理 + 没有判断
- 说得更顺
- 真正卡 + 选择
- 卡在 + 选择
- 只是 + 整理
- 不像 + 理解
- 没有 + 判断质量
- 伪理解

负例：

- 用户只是要求改写文案，不涉及系统判断质量。
- 用户要求摘要一段资料，不要求军师理解处境。
- 用户说输出风格不喜欢，但没有质疑判断链路。

信号类型：

```text
risk / judgment_quality
```

反方提醒：

```text
如果系统只能复述输入，哪怕链路完整，也只是高级表单和模板提醒。
```

建议动作：

```text
先评分事件理解、信号识别和反迎合能力；低于 4 分就补语义规则，不进 SQLite。
```

## Cluster 2：platform_competition_risk

语义本质：

用户担心系统级入口、分发入口或平台能力被大厂掌握，导致个人超级军师方向被覆盖或失去独立空间。

语义线索：

- 大厂 + 吃掉
- OpenAI
- Google
- Apple
- 平台 + 掌握
- 系统级入口
- 默认入口
- 入口 + 掌握
- 入口 + 模型厂商
- 长在 + 操作系统
- 模型厂商
- 分发入口
- 操作系统
- 还有没有必要
- 方向 + 继续做
- 占住 + 位置
- 被 + 覆盖
- memory + agent
- personal assistant

负例：

- 用户只是想了解某家公司的新闻，不涉及本项目方向判断。
- 用户只是比较 API 价格，不涉及平台入口风险。
- 用户提到 Apple 或 Google 作为普通数据源，不涉及竞争格局。

信号类型：

```text
risk / competition_risk
```

反方提醒：

```text
如果差异化只是 memory、agent、personal assistant，很容易被平台型公司覆盖。
```

建议动作：

```text
把焦虑转成差异化验证任务：护城河是否来自个人局势主循环、证据链、反迎合和长期结果追踪。
```

## Cluster 3：scope_creep_risk

语义本质：

用户在当前版本判断质量尚未验证前，又想扩展语音、浏览器、邮件、健康、位置等终局功能，导致验证任务被功能扩张稀释。

语义线索：

- 继续讨论 + 终局功能
- 又开始想
- 都加进去
- 语音
- 浏览器
- 邮件
- 健康数据
- 位置
- 判断质量 + 没有 + 验证
- 核心判断 + 没 + 证明
- 核心判断 + 没有 + 证明
- 更多输入来源
- 输入来源 + 全列
- 还没有测试
- 当前这一版 + 没有真正验证
- 先加 + 功能
- 扩展功能
- 数据接入

负例：

- 用户只是记录未来路线图，不打算改变当前验收顺序。
- 用户要求为未来功能写约束文档，而不是现在开发。
- 用户讨论数据源的权限边界，不影响当前测试任务。

信号类型：

```text
deviation / scope_creep
```

反方提醒：

```text
继续讨论语音、浏览器、邮件、位置、健康数据，会让你绕开当前 V1 判断质量问题。
```

建议动作：

```text
停止扩展功能讨论，先完成真实事件测试，并按分数决定是否继续补规则。
```

## Cluster 4：commitment_gate

语义本质：

用户明确设置阶段门槛：先完成测试、评分和规则修正，达标后才进入数据库或下一阶段开发。

语义线索：

- 先 + 测试
- 跑扎实
- 分数不够
- 继续改规则
- 不要急着 + 数据库
- 不马上 + SQLite
- 完成 + 真实事件测试
- 根据结果 + 修改判断规则
- 达到门槛
- 进入下一阶段
- 审计 + 之前
- 写死 + 持久化
- 固化 + 持久化
- 规则 + 写死
- 先不继续扩展
- 不急着做
- 门槛

负例：

- 用户只是表达希望以后测试，没有形成阶段门槛。
- 用户只是说数据库可以晚点做，但没有验收条件。
- 用户提出一个普通待办，不涉及阶段准入。

信号类型：

```text
commitment / commitment_tracking
```

反方提醒：

```text
说“先测试”不等于真的会停下扩展冲动，必须有分数门槛和复测记录。
```

建议动作：

```text
把承诺变成验收门槛：完成报告、补规则、复测达标后再决定是否进入 SQLite。
```

## 触达模板要求

关键 signal 的 touchpoint 必须包含：

```text
我观察到的信号
为什么重要
和当前阶段/长期目标的关系
反方判断
现在应该做什么
如果不做的后果
```

## 当前边界

这仍然不是模型级语义理解，而是“显式语义簇规则”。

它比单句关键词更稳，但仍需要持续用变体测试扩展：

- 同义表达
- 反向表达
- 低显性表达
- 多意图混合事件
- 负例误触发

## 防过拟合原则

1. 禁止为任何单条测试句创建专属规则。
2. 每个 semantic cluster 必须描述一类稳定语义，而不是覆盖某个句子。
3. 如果某条规则只能解释 1 条测试输入，视为过拟合，不允许计入有效补强。
4. 每个 semantic cluster 必须包含：
   - 本质定义
   - 正向语义线索
   - 负向样例
   - 非测试验证样例
   - 当前阶段下的触发条件
   - 不应触发的边界
5. 每轮复测必须包含固定测试集和 hidden semantic variants，防止只针对题库优化。

## Cluster 字段标准

每个 semantic cluster 至少包含：

```text
intent
indicators
negative_examples
signal_type
recommended_action
counter_argument
touch_template
```

建议补充：

```text
non_test_validation_examples
current_phase_trigger
boundary_not_trigger
```

当前静态原型先以 `intent`、`indicators`、`negative_examples` 和 `touch_template` 承载这些含义；后续进入 readiness review 前，需要把补充字段显式结构化。

## 非测试验证样例

judgment_quality_risk：

```text
这个回答像是把我的话整理了一遍，但没有判断我现在真正卡在哪里。
系统说得挺顺，但我看不出它为什么这么判断。
```

platform_competition_risk：

```text
如果未来入口都在操作系统和模型厂商手里，个人项目还能抓住什么位置？
平台一旦把个人助理做成默认能力，我们还有什么不同？
```

scope_creep_risk：

```text
我想先把更多数据源都列进来，但现在核心判断还没证明。
是不是应该先接更多输入，否则这个产品不够完整？
```

commitment_gate：

```text
先把评分门槛跑过，再讨论下一阶段。
没有过审计之前，不要把规则写死进持久化层。
```

## 当前阶段触发条件

当前阶段是 V1 最终测试审计阶段。

因此以下信号应提高优先级：

- 判断质量风险
- 固定测试过拟合风险
- hidden variants 失败风险
- 验证前扩张
- 进入 SQLite 前的阶段门槛

以下信号应降低优先级或不触发：

- 单纯 UI 美化想法
- 普通信息补充
- 不影响 V1 判断质量的远期功能设想
- 与当前主循环无关的灵感记录

## 不应触发的边界

- 用户只是做未来功能列表，不改变当前阶段动作，不应触发 scope creep。
- 用户只是询问某个平台新闻，不涉及项目是否继续，不应触发 platform competition risk。
- 用户只是要求改写表达，不质疑判断质量，不应触发 judgment quality risk。
- 用户只是记录普通待办，不涉及阶段门槛，不应触发 commitment_gate。
