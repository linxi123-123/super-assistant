# W7 日常输入字段规格

生成时间：2026-06-11

## 1. 字段总览

每日作战输入包建议字段：

1. `input_date`
2. `key_event`
3. `current_question`
4. `project_state_change`
5. `committed_action`
6. `constraints_or_forbidden`
7. `evidence_note`
8. `sensitivity`
9. `use_for_today_judgment`

其中前 6 个字段面向用户；后 3 个字段用于审计和边界判断。

## 2. `input_date`

用途：

记录输入属于哪一天。

是否必填：

是。

示例：

```text
2026-06-11
```

缺失降级：

使用当前日期，但必须标注“日期由系统推定”。

## 3. `key_event`

用途：

记录当天最重要的真实事件或事实变化。

是否必填：

至少应填写 `key_event` 或 `current_question` 之一。

示例：

```text
我发现作战室页面能看懂当前阶段了，但还不知道每天应该输入什么。
```

规则：

- 优先写事实，不写空泛情绪。
- 可以包含用户反馈。
- 不应自动升级为长期记忆。

## 4. `current_question`

用途：

记录用户今天最想让作战室帮助判断的问题。

是否必填：

至少应填写 `key_event` 或 `current_question` 之一。

示例：

```text
我明天应该先设计日常输入机制，还是先做记忆治理？
```

规则：

- 问题必须可判断。
- 不能只是闲聊。
- 应能映射到当前局势或今日动作。

## 5. `project_state_change`

用途：

记录项目、目标、关系或资源状态发生了什么变化。

是否必填：

否，但强烈建议填写。

示例：

```text
W6E 已通过，用户选择进入 W7。
```

规则：

- 用于更新当前阶段语境。
- 不直接写入长期记忆。

## 6. `committed_action`

用途：

记录用户已经承诺或准备今天完成的动作。

是否必填：

否。

示例：

```text
今天只完成 W7 日常输入机制规格，不做页面开发。
```

规则：

- 可以用于生成“今日唯一动作”。
- 如果多个动作冲突，应降级为待澄清。

## 7. `constraints_or_forbidden`

用途：

记录当前明确禁止事项或限制条件。

是否必填：

是。

示例：

```text
不改页面，不改 JSON，不接外部情报，不进入执行代理。
```

规则：

- 必须进入作战室第一屏的“不该做什么”或阶段门槛。
- 不能被历史模块覆盖。

## 8. `evidence_note`

用途：

记录输入的证据来源或用户说明。

是否必填：

否。

示例：

```text
来自用户在 W6E 后的明确选择。
```

规则：

- 用于审计。
- 不作为独立判断。

## 9. `sensitivity`

用途：

标记输入敏感度。

可选值：

```text
low / medium / high
```

是否必填：

是。

规则：

- `high` 输入不得自动进入长期记忆。
- `high` 输入不得用于外部传输。

## 10. `use_for_today_judgment`

用途：

标记该输入是否允许参与今日作战室判断。

可选值：

```text
yes / no / unclear
```

是否必填：

是。

规则：

- `yes`：可用于当前判断。
- `no`：只进入审计或记录。
- `unclear`：不得生成强判断，需用户澄清。
