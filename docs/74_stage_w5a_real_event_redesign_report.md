# W5A 真实事件作战室重构准备报告

## 1. 本阶段目标

W5A 的目标是基于用户真实反馈，重新定义个人作战室 V1 的信息层级，并准备一版基于真实事件的作战室验证方案。

本阶段不是开发阶段，不改页面，不改 JSON，不改 SQLite，不接新数据源。

## 2. 用户反馈摘要

用户反馈：

- 页面不够直观。
- 当前数据不是真实事件，无法验证真实作战价值。
- 页面设计不够明确。
- 页面整体更像列表 / dashboard，而不像个人作战室。

## 3. 为什么 W5 未通过

W5 未通过不是因为页面无法加载，也不是因为模块缺失。

W5 未通过的核心原因是：

- 页面缺少强主判断。
- 第一屏没有形成“军师简报”。
- 测试 case 缺乏真实紧迫感。
- 模块展示过于平均，用户难以一眼判断当前最重要事情。

## 4. 为什么测试 case 不足以验证作战室价值

测试 case 适合验证：

- 数据链路
- 信号提取
- 假设修正
- 审计字段
- 页面读取

但测试 case 不足以验证：

- 用户真实局势感
- 当前行动收敛
- 当天风险提醒
- 真实犹豫和走偏防护

作战室价值必须用真实事件验证。

## 5. 为什么当前页面像列表

当前页面模块完整，但每个模块权重接近。

它更像：

- dashboard
- 模块列表
- 审计报告页

而不像：

- 作战室
- 军师简报
- 当前局势面板

主要问题不是样式，而是信息层级。

## 6. 为什么下一步要用真实事件

真实事件能提供：

- 真实困惑
- 真实风险
- 真实行动状态
- 真实不确定性
- 真实阶段偏离

只有这些输入存在，作战室才能判断：

- 当前最大矛盾是什么
- 今日唯一动作是什么
- 什么不该做
- 如果继续错方向会怎样

## 7. 为什么暂不改页面

当前不应立刻改页面。

原因：

- 还没有真实事件快照。
- 第一屏新结构尚未用真实数据验证。
- 如果直接改页面，可能只是另一轮 UI 美化。
- 当前问题是产品表达与数据真实性，不是工程组件不足。

## 8. 下一阶段建议

建议进入：

W5B：真实事件快照生成准备。

W5B 应只允许：

- 收集用户真实事件输入
- 生成真实事件样例
- 设计或生成 `real_war_room_snapshot_v1.json`
- 校验真实快照是否能回答第一屏五个关键问题

W5B 不应进入：

- 外部情报
- 执行代理
- 新数据源
- 页面重构
- UI 美化

## 9. 是否触碰禁止事项

未触碰禁止事项。

本阶段未发生：

- 改页面
- 改 JSON
- 改 SQLite
- 写新数据
- 接新数据源
- 外部情报
- 执行代理
- npm / 框架
- 后端服务

## 10. 报告结论

当前不建议进入外部情报、执行代理或新数据源。下一步应进入 W5B：真实事件快照生成准备。
