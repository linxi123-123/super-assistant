# W7 阶段启动：真实作战室日常输入机制规格设计

生成时间：2026-06-11

## 1. 阶段定位

W7 是“真实作战室日常输入机制规格设计”阶段。

W7 不是页面开发阶段，不是数据接入阶段，不是外部情报阶段，不是执行代理阶段，也不是 M0 长期记忆治理中心实现阶段。

W7 的核心问题是：

```text
用户每天应该以什么最小方式输入真实事件，系统如何把这些输入变成可审计的作战室当前判断？
```

## 2. 进入依据

进入 W7 的依据：

- W6D 已完成第一屏当前语境最小页面改造。
- W6E 人工验收已由用户确认通过。
- 第一屏已经能区分当前判断与历史审计。
- 当前项目下一步需要从一次性真实事件快照，走向可持续的日常输入机制规格。

## 3. W7 目标

W7 只做规格设计，目标包括：

- 定义日常输入的最小字段。
- 定义日常输入和作战室当前判断之间的转换关系。
- 定义哪些输入可以进入当前判断，哪些只能作为历史审计。
- 定义日常输入的验收标准。
- 定义后续最小实现阶段的边界。

## 4. W7 必须回答的问题

W7 需要回答：

- 用户每天最少需要输入什么？
- 输入应该偏事件、状态、问题、决策，还是混合结构？
- 哪些输入属于“当前判断依据”？
- 哪些输入只能进入“历史记录 / 审计”？
- 如何避免日常输入退化成聊天记录？
- 如何避免用户每天填写过多字段？
- 如何避免系统在没有记忆治理中心前过度沉淀长期记忆？
- 日常输入如何服务作战室第一屏五个关键问题？

## 5. W7 允许事项

W7 允许：

- 日常输入机制 PRD。
- 日常输入字段规格。
- 日常输入到作战室判断的数据映射。
- 日常输入验收标准。
- W7 阶段报告。
- 后续最小实现边界草案。
- 更新 `tasks/MASTER_ROADMAP.md`。

## 6. W7 禁止事项

W7 禁止：

- 修改 app 页面。
- 修改 JSON 数据。
- 修改 SQLite schema。
- 写入 SQLite。
- 接入新数据源。
- 读取外部网络。
- 做外部情报雷达。
- 做执行代理。
- 进入 M0 记忆治理中心开发。
- 引入 npm / React / Vue / Next.js / Electron / 第三方库。
- 做后端服务。
- 新增聊天、编辑、保存、同步、上传等功能。

## 7. 与 M0 的关系

W7 和 M0 是不同方向：

- W7 解决“真实作战室每天如何获得输入”。
- M0 解决“长期记忆如何治理，防止几年尺度上混乱”。

W7 不应伪装成 M0，也不应提前实现长期记忆治理中心。

## 8. 当前结论

W7 已启动。下一步应输出日常输入机制的详细规格，而不是直接开发页面或接入数据源。
