# 长期记忆治理差距分析与路线图

生成时间：2026-06-11

参考图：长期个人军师：记忆治理与判断闭环

## 1. 长期架构核心层

这张图定义的长期个人军师系统不是一个聊天助手，也不是单一作战室页面，而是一套围绕“长期记忆治理”和“判断闭环”的系统结构。

### 1.1 感知与输入层

该层负责接收真实世界和用户自身状态的输入，包括：

- 真实事件：外部发生的事实与信息。
- 用户主动输入：想法、问题、计划、诉求。
- 历史行为与结果：过去的行动及结果数据。
- 项目、关系、目标状态：当前状态、进展、约束与变化。

这一层的价值不是堆数据源，而是把可解释、可追溯、和用户目标相关的信息送入系统。

### 1.2 事件理解层

该层负责把输入转成可处理的事件和候选判断，包括：

- 事件解析：拆解事实、理解背景与含义。
- 候选记忆提取：匹配相关记忆与历史经验。
- 个人模型假设：基于模型形成可能解释。
- 关键信号识别：提炼对判断最关键的信号。

这一层是从“输入材料”到“可判断事件”的转换层。

### 1.3 记忆治理中心

这是图中最关键的长期系统中枢。它不只是保存记忆，而是治理记忆。

图中定义的记忆治理中心包括四组能力：

- 记忆分级：事实、假设、偏好、承诺、情绪、判断。
- 每条记忆核心字段：来源事件、发生时间、置信度、重要性、适用范围、有效期、用户确认、反证、最后使用时间、是否允许用于建议。
- 治理机制：用户确认、周期复审、过期/修正/删除、建议引用追踪、禁止某条记忆参与建议、错误判断记录。
- 输出原则：少量高质量记忆、可审计、不把猜测当事实、旧判断不能污染新判断。

长期个人军师是否能用几年，主要取决于这一层是否成立。

### 1.4 推理与军师判断层

该层负责基于当前情境、长期目标、记忆治理结果和证据链给出判断，包括：

- 当前局势判断。
- 长期目标对账。
- 风险 / 机会识别。
- 反方判断。
- 今日唯一动作。
- 不要做清单。
- 显式读取自己的记忆治理中心。

这一层的核心要求是：判断必须能说明自己用了什么证据、哪些记忆、哪些反例，以及置信度边界。

### 1.5 输出与触达层

该层负责把判断转化为用户可理解、可行动、不过度打扰的输出，包括：

- 作战室第一屏。
- 主动提醒。
- 军师简报。
- 决策建议。

输出不应只是展示信息，而应帮助用户快速理解当前最重要的局势、动作和边界。

### 1.6 反馈与校正闭环

该层负责让系统从结果中学习，包括：

- 用户反馈：对建议 / 判断的评价与补充。
- 结果追踪：跟踪行动结果与关键指标。
- 模型修正：修正个人模型与判断逻辑。
- 记忆复审：复审相关记忆，更新字段与分级。
- 规则更新：更新治理规则、输出原则与策略。

没有反馈与校正闭环，系统会逐渐把旧判断、误判和噪声沉淀成“看似稳定的错误”。

## 2. 当前项目已完成部分

当前项目已经完成了长期系统的若干基础模块，但还没有完成长期记忆治理中心。

### 2.1 事件链路

项目已经跑通过最小主循环：

```text
事件 -> 记忆 -> 模型 -> 信号 -> 触达 -> 反馈 -> 修正
```

这证明系统已经不是单纯聊天页面，而是具备事件驱动的产品骨架。

### 2.2 历史快照

项目已经生成 `data/history_snapshot_v1.json`，并以 15 条历史 case 支撑历史审计、判断质量复盘和后续作战室数据映射。

### 2.3 SQLite 持久化

项目已经完成 SQLite 本地持久化、schema、初始化脚本和 smoke test，并完成多事件历史持久化测试。

这说明项目具备长期数据落地的早期基础，但当前尚未围绕长期记忆治理扩展 schema 或业务规则。

### 2.4 历史审计面板

项目已经完成只读历史审计面板，并补强了：

- `phase_context`
- `evidence_chain`
- `revision_explanation`
- `audit_readiness`

历史面板已经能帮助审计判断质量，但它仍偏向历史 case 展示与审计，不是长期记忆治理中心。

### 2.5 作战室第一屏

项目已经完成个人作战室第一屏的最小改造，能够优先回答五个问题：

- 当前局势一句话。
- 当前最大矛盾。
- 军师直接判断。
- 今日唯一动作。
- 不要做清单。

这提升了“打开页面后快速理解局势”的能力。

### 2.6 真实事件快照

项目已经基于真实事件生成 `data/real_war_room_snapshot_v1.json`，并明确区分真实事件快照与旧测试快照。

这一步让作战室从纯测试 case 走向真实输入验证，但还不是长期持续数据系统。

### 2.7 证据链 / 反方判断 / 今日动作 / 不要做清单

当前项目已经持续强调：

- 判断需要证据链。
- 判断需要反方视角。
- 页面需要给出今日唯一动作。
- 页面需要明确不要做什么。

这些能力与图中的“推理与军师判断层”高度一致，是长期个人军师价值的核心基础。

## 3. 当前项目关键缺口

当前最大缺口不是外部能力，而是记忆治理中心。

### 3.1 记忆分级

当前项目已有候选记忆和记忆类型，但还没有形成稳定的长期分级制度：

- 事实
- 假设
- 偏好
- 承诺
- 情绪
- 判断

缺少分级会导致系统把不同性质的信息混在一起，例如把一次情绪表达当成长期事实，把一次猜测当成稳定判断。

### 3.2 用户确认机制

项目已有 `user_confirmed` 方向，但还缺少完整的确认工作流。

长期记忆不应自动进入核心模型。尤其是敏感、长期、会影响建议的记忆，必须经过用户确认、修改或拒绝。

### 3.3 长期记忆治理字段

图中要求每条记忆具备一组治理字段。当前数据模型已有部分字段，但还缺完整的规格约束与使用规则。

需要明确：

- 来源事件如何追溯。
- 有效期如何设置。
- 反证如何进入。
- 最后使用时间如何记录。
- 是否允许用于建议如何执行。
- 适用范围如何限制。

### 3.4 记忆过期机制

当前项目尚未形成明确的过期策略。

长期个人军师不能默认所有旧记忆永久有效。人的目标、状态、关系、项目阶段都会变化。旧记忆如果不过期，会干扰新判断。

### 3.5 记忆复审机制

当前项目还缺少周期性记忆复审机制。

长期系统至少需要支持：

- 周复审：检查近期高影响记忆。
- 月复审：检查长期目标、承诺和高置信判断。
- 阶段复审：项目阶段变化时复审相关记忆。

### 3.6 建议引用了哪些记忆

当前项目强调证据链，但还需要进一步明确每条建议引用了哪些记忆。

长期系统必须能回答：

```text
这条建议是基于哪些事件、哪些记忆、哪些模型假设得出的？
```

否则用户无法判断建议是否被旧记忆污染。

### 3.7 禁止某条记忆参与建议

长期记忆系统必须允许用户把某条记忆设为：

```text
保留记录，但禁止用于建议。
```

这和删除不同。删除是移除记录；禁止参与建议是保留审计，但不允许它影响未来判断。

### 3.8 错误判断记录

项目已有模型修正和反馈方向，但还需要明确“错误判断记录”的一等地位。

真正的长期军师必须记得自己错在哪里，而不是只记得用户说过什么。

错误判断记录应包含：

- 原判断。
- 当时证据。
- 当时置信度。
- 反证或结果。
- 错误原因。
- 修正规则。
- 后续避免方式。

### 3.9 周期性复盘

当前项目尚未形成周期性复盘制度。

长期个人军师需要定期对账：

- 长期目标是否变化。
- 当前行动是否偏离目标。
- 哪些建议有效。
- 哪些判断误导。
- 哪些记忆需要过期、降权或删除。

### 3.10 新旧判断冲突检测

当前项目尚未形成新旧判断冲突检测。

几年尺度上，冲突一定会出现。例如：

- 旧目标与新目标冲突。
- 旧偏好与当前状态冲突。
- 旧承诺与新行动冲突。
- 旧判断与新证据冲突。

系统必须能把冲突显式暴露出来，而不是默默平均或选择性引用。

## 4. 风险判断

### 4.1 如果现在直接加外部情报

如果当前阶段直接进入外部情报，会有三类风险：

- 噪声放大：外部信息会显著增加输入量，但当前缺少稳定记忆治理，容易把无关信息误判为高价值信号。
- 目标漂移：外部趋势、新闻、机会会持续诱导系统偏离用户真实目标。
- 证据链污染：外部材料如果没有和用户记忆、目标、阶段严格对账，会让作战室看似更丰富，实际更不可信。

因此，在记忆治理中心没有规格化之前，不建议进入外部情报。

### 4.2 如果现在直接加执行代理

如果当前阶段直接进入执行代理，会有更高风险：

- 错误判断会被放大成真实行动。
- 旧记忆或未确认假设可能触发错误执行。
- 用户授权边界尚未和长期记忆治理绑定。
- 系统可能在缺少复审、反证和确认机制时自动推进任务。

执行代理必须建立在可审计判断、权限边界、记忆引用和结果追踪之上。当前阶段不建议进入执行代理。

### 4.3 如果现在没有记忆治理就长期使用

如果没有记忆治理就直接长期使用，主要风险包括：

- 旧记忆干扰新判断。
- 把猜测当事实。
- 噪声越来越多。
- 系统越用越自信、越用越偏。
- 用户难以知道某条建议到底受哪些旧信息影响。
- 错误判断没有被记录和纠正，反而沉淀成长期模型。

长期使用的关键不是“记得更多”，而是“治理记忆、审计判断、持续校正”。

## 5. 下一阶段建议

不建议立即实现整张图。

这张图可以作为长期架构参考，但它覆盖了输入、理解、记忆治理、推理、输出、反馈、规则更新等完整体系。如果直接按图开发，会过早进入大系统实现，偏离当前阶段的最小验证路线。

当前下一阶段建议只在两个方向中选择其一：

### 5.1 W6：第一屏当前语境与阶段时效规格修正

如果继续沿作战室路线，建议进入 W6。

W6 目标是修正 W5E 发现的第一屏规格问题：

- 增加当前阶段标识。
- 区分历史动作和当前动作。
- 为军师直接判断增加时效标签。
- 压缩解释层。
- 明确旧模块与第一屏判断的证据关系。

W6 仍应是规格修正，不应直接扩展外部能力。

### 5.2 M0：长期记忆治理中心规格设计

如果转向长期可用性，建议进入 M0。

M0 目标是只做“长期记忆治理中心”的规格设计，包括：

- 记忆分级规则。
- 每条记忆核心字段。
- 用户确认机制。
- 过期 / 修正 / 删除机制。
- 周期复审机制。
- 建议引用追踪。
- 禁止记忆参与建议。
- 错误判断记录。
- 新旧判断冲突检测。
- 与作战室、历史审计、SQLite 的边界关系。

M0 不应开发页面，不应改 SQLite，不应写新数据。

## 6. M0 边界

如果进入 M0，M0 只允许做规格设计，不允许开发。

M0 允许事项：

- 记忆治理中心 PRD。
- 记忆分级规格。
- 记忆字段规格。
- 记忆复审工作流。
- 建议引用与审计规则。
- 错误判断记录规格。
- 新旧判断冲突检测规格。
- 后续开发任务拆解。

M0 禁止事项：

- 不改 app 页面。
- 不改 SQLite。
- 不写新数据。
- 不接新数据源。
- 不进入外部情报。
- 不进入执行代理。
- 不引入 npm / 框架 / 第三方库。
- 不把架构图直接转成开发任务。

## 7. 结论

这张图可以作为长期个人军师系统的方向参考，但不能直接作为当前开发任务。

当前项目已经具备事件链路、历史快照、SQLite 持久化、历史审计面板、作战室第一屏、真实事件快照、证据链、反方判断、今日动作和不要做清单等基础能力。

但当前项目离“可长期使用几年”的个人军师还缺一个关键中枢：记忆治理中心。

因此，下一步不应继续扩外部能力，也不应直接进入执行代理。更合理的路线是：

```text
先补记忆治理中心规格
再让作战室显式读取治理后的记忆
再考虑外部情报
最后才考虑执行代理
```

当前最稳妥的判断是：长期方向成立，但必须先补“记忆治理中心”，否则长期使用会放大记忆混乱和判断偏差。
