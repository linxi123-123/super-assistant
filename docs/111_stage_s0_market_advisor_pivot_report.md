# S0 阶段报告：市场军师最小可用闭环规格与原型

生成时间：2026-06-11

## 1. 为什么转向市场军师最小闭环

用户反馈表明，当前个人军师做了很多页面、文档、审计和快照，但真实可用性不足。

用户真正期待的是：问一个现实问题，系统能给出有用判断。

“今天股市怎么样”就是一个高频、具体、可验证的问题。

## 2. 当前项目为什么让用户觉得没用

原因包括：

- 过度重视架构和流程。
- 缺少真实外部数据。
- 缺少用户持仓 / 关注列表。
- 缺少真实日常问题闭环。
- 作战室页面仍不能解决用户高频问题。

## 3. 真正有用的个人军师需要哪些数据

市场军师至少需要：

- A 股、港股、美股市场概况。
- 用户关注标的。
- 用户确认持仓。
- 标的风险备注。
- 用户投资 / 观察风格。
- 数据来源和时间戳。

## 4. 当前先不接外部行情时能做什么

在不接外部 API 的情况下，可以先做：

- 用户手动维护市场画像。
- 用户手动粘贴行情材料。
- 系统识别市场问题。
- 系统基于用户提供材料和本地画像输出结构化判断。
- 系统明确数据边界，不假装实时。

## 5. 用户市场画像如何设计

已定义：

- `watchlist`
- `holdings`
- `market_preferences`

原则：

- 不接券商账户。
- 不要求输入敏感金额。
- 成本和仓位 optional。
- 所有持仓必须用户手动确认。
- 系统不能猜测用户持仓。

## 6. 市场问题回复如何设计

市场问题回复必须包含：

- 查询类型。
- 数据状态。
- 市场范围。
- 简短回答。
- A 股 / 港股 / 美股摘要。
- 关注标的相关性。
- 持仓相关性。
- 军师判断。
- 风险提醒。
- 下一步检查事项。
- 不要做什么。

## 7. 后续 S0D 最小开发边界

S0D 如果启动，只允许：

- 识别市场类问题。
- 读取本地用户市场画像。
- 基于用户粘贴材料做简短分析。
- 显示关注列表和持仓相关提醒。
- 不接外部 API。
- 不接券商账户。
- 不写 SQLite。
- 不引 npm / 框架。

## 8. 风险与禁止事项

禁止：

- 自动交易。
- 预测涨跌。
- 伪造实时行情。
- 编造用户持仓。
- 编造新闻。
- 直接推荐买入 / 卖出。
- 接券商账户。
- 接外部 API。

## 9. 是否建议进入 S0D

建议进入 S0D，但必须由用户单独确认。

原因：

市场军师最小闭环能最快验证“个人军师是否真实有用”。

## 10. 报告结论

当前不应继续堆作战室页面和文档，应优先验证市场军师最小闭环是否能让用户感到真实有用。是否进入 S0D 最小开发，需要用户单独确认。
