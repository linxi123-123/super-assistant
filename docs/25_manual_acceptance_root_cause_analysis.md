# 人工验收低分根因分析

## 1. 人工评分低的事实

阶段 O 人工验收结果：

- 平均分：18/30
- 每个维度平均：3/5
- 15 条全部需优化
- 通过：0
- 不通过：0

这说明当前历史面板不是失败状态，但还没有达到“可以支撑个人作战室 V1”的审计质量。

## 2. 低分原因分类

### A. 数据是否缺失

检查项：

- `revision_reason`：存在，但部分内容偏通用，且有些来自旧对象或文本摘要，不是统一结构化解释。
- `old_confidence / new_confidence`：存在。
- `feedback_type`：存在。
- `outcome`：存在。
- `evidence`：存在，但证据链分散在 event、memory、hypothesis、signal、touchpoint 中。
- `counter_evidence`：存在于 hypothesis，但人工阅读时不够突出。
- `related_phase / currentPhaseContext`：`related_phase` 存在，`currentPhaseContext` 不作为独立审计对象显式呈现。
- `recommended_action`：存在。
- `consequence_if_ignored`：存在。

判断：

数据不是完全缺失。主要问题不是 SQLite schema 缺字段，而是现有字段没有形成足够清晰的审计解释层。

### B. 展示是否不足

检查项：

- 修正理由：存在，但需要进入详情阅读，且不够结构化。
- 为什么置信度上调/下调：有 old/new confidence，但原因没有被单独高亮。
- 当前阶段上下文：有 `related_phase` 和 `phase_relation`，但用户难以一眼看出阶段目标、禁止事项和该 signal 与阶段的关系。
- 证据链：字段存在，但分散，阅读成本高。
- 反方判断：存在，但摘要层不够明显。
- 行动建议和后果：存在，但在列表层不可见。

判断：

展示层确实不足。不是 UI 美化问题，而是审计字段的可见性和组织方式不足。

### C. 解释是否不够具体

检查项：

`model_revision.revision_reason` 当前常见问题：

- 有些是 JSON 字符串，人工阅读成本高。
- 有些只是“置信度上调”式摘要。
- 不总是明确说明原假设、用户反馈支持了哪一部分、为什么调整置信度、后续如何验证。

判断：

解释生成规则不足是核心瓶颈之一。人工备注中“模型修正理由可以更具体”正好指向这一点。

### D. 阶段上下文是否没有被显式呈现

检查项：

每条 case 目前能看到：

- 当前阶段字段：部分存在，如 `related_phase = Stage L`。
- 当前阶段目标：不稳定，通常藏在 touchpoint 文本里。
- 当前阶段禁止事项：没有作为结构化字段出现。
- signal 与当前阶段关系：部分由 `phase_relation` 表达，但过于通用。
- recommended_action 为什么适合当前阶段：通常能推断，但不够显式。

判断：

阶段上下文可审计性不足。用户能看到链路，但不一定能快速判断“为什么这个提醒在当前阶段重要”。

## 3. 结论

主要瓶颈是多个因素叠加：

- 数据缺失：不是主因。核心字段大多存在。
- 展示不足：是重要原因。证据链、修正理由、阶段上下文没有形成清晰审计视图。
- 解释规则不足：是主要原因。`revision_reason` 不够结构化、不够具体。
- 阶段上下文缺失：是主要原因。当前阶段目标、禁止事项、signal 关系没有显式呈现。

最终判断：

```text
低分不是因为页面不好看，而是因为历史面板还不能让用户足够轻松地审计“为什么系统这样修正模型”和“为什么这个判断适合当前阶段”。
```

因此不建议进入个人作战室 V1。应先做历史面板审计补强。
