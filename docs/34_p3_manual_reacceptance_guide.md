# P3 人工重新验收说明

## 1. 本阶段目标

阶段 P3 的目标不是继续开发，而是基于 P2 已增强的历史面板，重新组织人工验收。

用户需要逐条检查 15 条 case，判断新增的 `phase_context`、`evidence_chain`、`revision_explanation`、`audit_readiness` 是否真正提升了系统的可审计性和军师价值。

核心问题是：

增强后的历史面板，是否足以支撑用户判断这个系统有资格进入“个人作战室 V1 规格设计”。

## 2. 为什么要重新人工验收

上一轮人工验收主要看基础链路是否存在：

事件 -> 记忆 -> 假设 -> 信号 -> 触达 -> 反馈 -> 结果 -> 模型修正。

但上一轮暴露出一个关键问题：链路存在不等于判断可靠。用户仍然需要知道：

- 判断发生在什么阶段背景下
- 判断依据来自哪些证据
- 模型为什么被修正
- 这条 case 是否已经具备人工审计条件

P2 已把这些字段展示到历史面板，因此 P3 要重新验收：这些新增字段是否让用户更容易发现好判断、坏判断、伪反迎合、证据不足和模型修正不合理。

## 3. 用户需要打开的页面

请打开：

`http://127.0.0.1:8766/app/history.html`

建议操作方式：

1. 从 Case List 逐条点击 15 条 case。
2. 对照 Case Detail 中的新增审计字段。
3. 把人工评分和备注填写到 `docs/37_p3_case_review_draft.md`。
4. 最后把 15 条结果汇总到 `docs/36_p3_manual_reacceptance_summary_template.md`。

## 4. 本轮重点查看的新增字段

### phase_context

重点判断：

- 是否说明当前阶段目标
- 是否说明本阶段禁止事项
- 是否解释这条 case 为什么在当前阶段重要
- 是否帮助用户避免把工程进展误认为军师能力达标

### evidence_chain

重点判断：

- 是否能从输入追踪到事件、假设、信号、触达、反馈、结果和模型修正
- 是否能解释这条判断为什么产生
- 是否能帮助用户发现证据不足
- 是否只是机械罗列字段

### revision_explanation

重点判断：

- 是否解释模型置信度为什么变化
- 是否说明用户反馈对假设的影响
- 是否包含后续验证计划
- 是否能帮助未来判断继续修正，而不是只做一次性记录

### audit_readiness

重点判断：

- 是否指出这条 case 是否具备审计条件
- 是否列出 audit blockers
- 是否能帮助用户快速判断哪些 case 可以信任、哪些需要修正
- 是否避免用高分掩盖缺陷

## 5. 本轮评分和上一轮的区别

上一轮主要看基础链路是否存在。

本轮重点看：

- 阶段上下文是否让判断更清楚
- 证据链是否让判断更可解释
- 模型修正解释是否更具体
- `audit_readiness` 是否帮助审计
- 页面是否能支撑进入个人作战室 V1

换句话说，P3 不是重新证明“系统会生成链路”，而是验证“用户能不能凭这些链路判断军师是否可靠”。

## 6. 进入下一阶段门槛

只有满足以下条件，才建议进入“个人作战室 V1 规格设计”：

- 15 条人工平均分 `>= 24/30`
- 判断可解释性平均 `>= 4/5`
- 军师提醒价值平均 `>= 4/5`
- 反迎合质量平均 `>= 4/5`
- 模型修正合理性平均 `>= 4/5`
- 页面可审计性平均 `>= 4/5`
- 不存在严重伪反迎合
- 不存在高价值信号沉默
- 不存在明显模型修正不合理
- 用户确认历史面板确实能帮助审计军师判断

## 7. 如果不达标

如果 P3 不达标：

- 不得进入个人作战室 V1
- 必须进入 P4：审计字段或提醒质量修正

P4 可能方向包括：

- P4 审计字段修正
- P4 touchpoint 质量修正
- P4 model_revision 解释修正
- P4 证据链结构修正
