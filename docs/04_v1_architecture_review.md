# V1 架构验收与补强

## 结论

当前 V1 静态原型已经从“演示式作战室”补强为“军师主循环的最小真实闭环”。

它仍然不是完整产品，也没有接入真实多源数据、模型 API 或数据库，但现在已经具备以下闭环：

```text
手动输入
-> 原始事件
-> 候选记忆
-> 个人模型假设
-> 信号识别
-> 主动触达
-> 用户反馈
-> 结果追踪
-> 模型修正历史
```

## 当前已经实现的环节

### 1. 手动输入

已实现。

位置：

- `app/index.html`
- `app/app.js`

用户可以输入今日事件、事件类型、重要性、关联目标、敏感度和原始证据。

### 2. 原始事件

已实现最小结构。

事件会保存在浏览器 localStorage 中，包含：

- 类型
- 内容
- 目标
- 重要性
- 置信度
- 敏感度
- 证据摘要

不足：

- 还没有真实 `occurred_at` 和 `captured_at` 分离。
- 还没有多来源数据同步。

### 3. 候选记忆

已实现。

保存事件后，系统会自动从高重要性事件、判断事件、承诺事件、风险事件和偏离事件中提取候选记忆。

每条候选记忆包含：

- 来源事件
- 证据
- 置信度
- 重要性
- 敏感度
- 是否确认

### 4. 个人模型假设

已实现最小版本。

系统根据判断事件、承诺事件、风险事件生成个人模型假设。

每条假设包含：

- 内容
- 证据
- 反例
- 验证计划
- 置信度

### 5. 信号识别

已实现最小版本。

当前支持：

- 承诺信号
- 风险信号
- 机会信号
- 状态信号
- 模式信号

每个信号包含：

- 证据
- 重要性
- 紧急性
- 可行动性
- 打扰成本
- 是否触达

### 6. 主动触达

已实现。

系统会从高价值信号生成军师提醒。提醒包含：

- 为什么提醒
- 观察到什么
- 风险/机会是什么
- 建议动作

### 7. 用户反馈

已实现。

用户可以对触达点击：

- 准确
- 不准确
- 稍后验证

反馈会保存到 `feedback` 状态中。

### 8. 结果追踪

已实现最小版本。

每条触达都会生成一个 outcome，用来记录预期结果、实际结果和结果状态。

目前结果追踪仍是轻量记录，不是完整复盘系统。

### 9. 模型修正

已实现最小版本。

当用户反馈：

- 准确：提高相关假设置信度。
- 不准确：降低相关假设置信度，并写入模型修正历史。
- 稍后验证：保持结果追踪为待验证。

## 哪些只是静态模拟

### 1. 多源输入

当前只有手动输入，不是真正多源。

尚未接入：

- 日历
- 任务系统
- 文件变动
- 浏览器记录
- 邮件/消息摘要
- Codex/Git 项目记录
- 健康/睡眠数据
- 会议录音

### 2. 事件抽取

当前事件类型由用户选择，不是模型自动抽取。

后续需要模型把自由文本拆成：

- 事实
- 判断
- 承诺
- 担忧
- 状态
- 机会
- 风险

### 3. 记忆提取

当前记忆提取是规则模拟。

后续需要模型判断：

- 是否值得长期保存
- 是否敏感
- 是否需要确认
- 是否存在反例
- 是否影响建议

### 4. 个人模型

当前个人模型是假设模板，不是深度推理。

后续需要基于历史事件、记忆、反馈和结果动态更新。

### 5. 信号识别

当前信号评分是规则模拟。

后续需要更细的评分逻辑，并避免重复触达和低价值打扰。

### 6. 主动触达

当前触达只显示在页面里，不是真正的定时提醒。

后续需要触达频率控制和渠道控制。

## 当前缺失或仍弱的环节

### 1. 自动事件抽取

缺失。

如果不补，系统会变成用户手动填表工具，无法成为真正军师。

### 2. 记忆确认和删除操作

部分缺失。

当前有候选记忆，但还没有在页面里提供确认、修改、删除、禁止用于建议的完整操作。

如果不补，记忆审计会停留在文档里，长期信任会不足。

### 3. 结果复盘

弱。

当前只记录反馈结果，没有让用户在后续补充“实际发生了什么”。

如果不补，系统无法验证判断是否正确，只能根据即时反馈修正。

### 4. 模型修正版本化

已补最小版本，但还不够细。

后续需要每次修正明确：

- 哪个假设被修正
- 为什么修正
- 哪些证据变化
- 哪些反例增加
- 置信度如何变化

### 5. 多源状态感知

缺失。

如果长期只靠用户主动输入，系统会偏向用户主观叙述，无法形成高质量局势判断。

### 6. 情报雷达

静态原型未实现。

如果不补，系统只能懂用户，不能懂世界，军师能力会被限制在内省和复盘。

## 不补会导致退化的关键点

### 1. 没有事件层

会退化成聊天记录堆积。

当前已实现事件层，后续必须保留。

### 2. 没有反馈闭环

会退化成单向建议工具。

当前已补反馈和模型修正历史，后续必须升级为数据库持久化。

### 3. 没有结果追踪

会退化成“说得像军师，但不验证判断”的工具。

当前只做了最小 outcome，后续必须补复盘入口。

### 4. 没有记忆审计

会退化成不可信的黑箱记忆。

下一步必须做记忆确认、修改、删除、禁用建议。

### 5. 没有模型修正历史

会退化成静态用户画像。

当前已补最小修正日志，后续必须做版本化。

### 6. 没有多源输入

会退化成高级日记。

V1 可以手动输入，但 SQLite 版本必须为日历、任务、文件和 Codex 项目记录预留采集表。

## 下一步验收标准

进入 SQLite 本地应用前，静态 V1 必须通过以下手动测试：

1. 录入一条判断事件。
2. 页面生成事件卡片。
3. 自动生成候选记忆。
4. 自动生成个人模型假设。
5. 自动生成信号。
6. 自动生成军师提醒。
7. 点击“不准确”。
8. 反馈页出现反馈记录。
9. 模型修正历史出现置信度变化。
10. 刷新页面后数据仍存在。
