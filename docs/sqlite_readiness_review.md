# SQLite Readiness Review

## 1. 为什么现在可以准备进入 SQLite

当前 V1 已经通过固定测试、语义变体测试和 hidden semantic variants 测试。

关键指标：

- 固定 10 条平均：27.15/30
- 变体 5 条平均：27.0/30
- Hidden 5 条平均：26.8/30
- 全部 15 条平均：27.03/30
- 信号识别：4.8/5
- 反迎合能力：4.87/5

这说明当前主循环已经具备最低判断质量：

```text
事件 -> 记忆 -> 模型假设 -> 信号 -> 触达 -> 反馈 -> 结果追踪 -> 模型修正
```

进入 SQLite 的理由不是“功能该升级了”，而是：

```text
现在已有足够理由把通过测试的主循环数据持久化，支持跨会话、跨天和多事件历史测试。
```

## 2. 当前哪些能力已通过测试

已通过：

- 重要行为事件能生成进展信号。
- 判断质量风险能识别为 `judgment_quality_risk`。
- 平台竞争风险能识别为 `platform_competition_risk`。
- 范围蔓延能识别为 `scope_creep_risk`。
- 阶段门槛能识别为 `commitment_gate`。
- 高价值信号能生成主动触达。
- 主动触达包含反方判断、阶段关系、建议动作和不行动后果。
- 用户反馈能触发模型修正。
- 模型修正能归因到对应假设。

## 3. 哪些能力仍然只是 V1 规则层，不应过度产品化

仍然只是规则层：

- semantic cluster 匹配。
- 信号评分。
- 记忆提取。
- 个人模型假设生成。
- 触达模板。
- 反馈后的置信度调整。

这些能力可以持久化记录，但不应包装成“已经懂用户”的最终能力。

## 4. SQLite 要固化哪些数据

优先固化：

- 原始事件 `events`
- 候选记忆 `memories`
- 个人模型假设 `personal_model_hypotheses`
- 信号 `signals`
- 触达 `touchpoints`
- 用户反馈 `user_feedback`
- 结果追踪 `outcomes`
- 模型修正历史 `model_revisions`

这些数据对应军师主循环的骨架。

## 5. 哪些数据暂时不应该固化

暂不固化：

- 浏览器历史
- 邮件
- 语音
- 健康数据
- 位置数据
- 任何 P2/P3/P4 高敏感数据
- 外部情报自动抓取结果
- 自动执行动作

原因：

当前阶段只验证了手动输入和规则判断质量，尚未验证多源数据权限、敏感度和噪音控制。

## 6. 进入 SQLite 后最大的走偏风险

最大风险：

```text
退化成 CRUD 系统。
```

具体表现：

- 只是在数据库里增删改查事件和记忆。
- 首页变成表格管理器。
- 信号和触达变成普通通知。
- 反馈和模型修正没有被真正用于下一轮判断。
- 记忆只保存，不审计、不过期、不验证。

## 7. 如何防止 SQLite 阶段退化成 CRUD 系统

必须坚持：

1. 每张表都对应主循环环节。
2. 每次写入都要说明服务哪个判断。
3. 每条记忆都能追溯来源事件。
4. 每条 signal 都有评分和触达理由。
5. 每条 touchpoint 都能接受反馈。
6. 每次反馈都要能更新 outcome 或 model_revision。
7. 首页仍是作战室，不是数据表入口。
8. SQLite 阶段完成后必须用同样 15 条测试回归。

## 结论

可以准备进入 SQLite。

但本文件不是开发授权。

下一步必须等待用户明确确认，才允许进入 SQLite 本地持久化阶段。
