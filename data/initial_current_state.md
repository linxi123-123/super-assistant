# 初始局势快照

生成时间：2026-06-08

## 当前阶段判断

用户正在从“想做一个个人超级助理”的模糊愿景，推进到“个人超级军师 / Jarvis 系统”的产品定义阶段。

当前关键任务不是马上写完整应用，而是把终局蓝图转成可开发、可验收、可防走偏的工程规格。

## 当前核心目标

构建一个属于用户个人的超级军师系统：

- 能长期理解用户。
- 能通过结构化记忆建立动态个人模型。
- 能读取全球前沿信息并判断和用户的关系。
- 能在关键时刻给出理性建议。
- 后续逐步具备执行能力，接近 Jarvis。

## 关键判断

1. 产品不能退化成“聊天框 + 记忆库 + 周报”。
2. 第一版可以小，但必须保留军师主循环。
3. 军师能力上限取决于状态感知质量。
4. 长期记忆不是聊天记录，也不是单纯向量库，而是服务未来判断的结构化记忆。
5. 系统必须主动，但不能制造噪音。
6. 系统必须能反驳用户，而不是迎合用户。
7. 执行能力必须建立在目标、权限和复盘机制之上。

## 当前主要风险

1. 过早进入开发，导致产品形态滑向普通 AI 助手。
2. 无限讨论终局，迟迟不形成可执行任务。
3. 把“不妥协”误解为“一开始做全部功能”。
4. 没有权限和记忆审计，导致系统越懂用户越难被信任。

## 当前最优动作

完成阶段 0 的工程化定稿：

- 产品宪法
- V1 PRD
- 主循环设计
- 数据模型
- 引擎规格
- 权限体系
- UI 信息架构
- Codex 开发任务链路

完成后进入阶段 1：

构建本地可运行的“个人局势军师 V1”。

## 初始个人模型假设

### 假设 1

用户目前不是缺少想法，而是需要把终局愿景转成可执行架构。

证据：

- 用户持续追问产品本质、长期记忆、状态感知和执行链路。
- 用户明确要求“完整方案和步骤链路”。
- 用户提供了终局蓝图，并要求开始执行。

置信度：0.86

反例：

- 用户此前也在推进其他具体项目，说明并非只停留在概念层。

验证计划：

- 未来 24-48 小时是否能从文档规格进入本地原型开发。

### 假设 2

用户对“普通 MVP 妥协”高度警惕，更看重主循环正确。

证据：

- 总纲反复强调“第一版可以小，但主循环不能错”。
- 用户担心产品退化成聊天助手。

置信度：0.9

反例：

- V1 仍需要范围控制，否则无法落地。

验证计划：

- V1 是否能接受较小功能范围，同时坚持事件、记忆、模型、信号、触达主循环。

## 初始候选记忆

1. 用户正在探索“个人超级军师 / Jarvis 系统”方向。
2. 用户希望系统不是聊天助手，而是持续运行的个人局势系统。
3. 用户认为长期记忆必须结构化，并且要有证据链、置信度、敏感度和确认机制。
4. 用户关心系统如何真实了解每天发生了什么，而不是只依赖主观日记。
5. 用户希望第一阶段先达成超级军师能力，第二阶段再升级为执行体。
6. 用户偏好真实、理性、可反驳的建议，不希望系统只迎合自己。
